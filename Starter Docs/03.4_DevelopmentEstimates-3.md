THE POINT
Claude Code does not replace the developer. It eliminates hours spent writing boilerplate, looking up syntax, and translating specifications into first-draft code. The developer reviews output, runs tests, makes judgment calls, and integrates modules. The same developer who would need 4-5 months traditionally can deliver this in 6-8 weeks.


This document breaks down every deliverable in the NovaPredict build into specific time estimates under two scenarios: traditional development (a skilled developer working from specs alone) versus Claude Code-assisted development (the same developer using Claude Code to generate first-draft code from the engineering prompts in the Dev Stack document). All estimates assume a solo developer. All hours are working hours, not calendar time.

HOW CLAUDE CODE WORKS
The developer opens their terminal, runs 'claude', and pastes the engineering prompt for a module. Claude Code reads the project structure, writes the complete module with tests, and the developer reviews, runs the tests, and integrates. For well-specified modules like the ones in this project, Claude Code produces production-ready first drafts in minutes rather than hours.

How to Read This Document


Each section covers one layer of the NovaPredict technical stack. Every deliverable shows:

	•	Traditional hours: realistic estimate for a skilled developer writing from specification with no AI assistance. Includes research, writing, debugging, and basic testing.
	•	Claude Code hours: realistic estimate using Claude Code to generate first drafts from the engineering prompts, with the developer reviewing, running tests, fixing edge cases, and integrating.
	•	Hours saved and reduction percentage.

WHAT CLAUDE CODE DOES BEST
Boilerplate-heavy modules, mathematical functions with clear specifications, database schemas, API integration code, test generation, and configuration files. These make up the bulk of this project. Savings are largest in the computation and data layers where every function is precisely specified.

WHAT CLAUDE CODE CANNOT DO
Architecture decisions, debugging novel integration issues, performance tuning, AWS infrastructure configuration, App Store submission, and client-side UX judgment calls. Developer expertise is irreplaceable in these areas. Claude Code hours are not near zero anywhere in this document.

What Claude Code Looks Like Day-to-Day


For a developer who has not used Claude Code before, here is exactly what the workflow looks like for a typical module in this project.

Step
Action
What Happens
1
Open terminal in project directory
Navigate to the relevant module folder. Run 'claude' to start a Claude Code session.
2
Paste the engineering prompt
Copy the full Claude Code prompt from [03.5_DevStackAndClaudeCodePrompts.md](03.5_DevStackAndClaudeCodePrompts.md). The prompt is the complete specification.
3
Claude Code reads the project
Claude Code scans the existing file structure, understands the context, and writes the complete module - the .py file and the test file. This takes 60-120 seconds.
4
Review the output
The developer reads through the generated code and checks that the logic matches the specification. This takes 15-30 minutes for a typical module.
5
Run the tests
Run 'pytest test_module.py -v'. All tests should pass if the specification was clear. Failed tests show exactly what needs fixing.
6
Fix edge cases
The developer fixes any test failures or logic gaps. For well-specified modules like the computation engine, this is typically minor.
7
Integrate
The reviewed module is integrated into the Lambda function or workflow. The developer handles integration - Claude Code does not do this step.
8
Commit
Commit with the prompt version number in the commit message for traceability.

The developer's expertise is fully engaged at steps 4, 5, 6, and 7. Claude Code handles steps 2 and 3. A module that would take a full day to write from scratch takes 2-3 hours with Claude Code - and comes with a complete test suite.

THE HONEST PITCH
You are not being asked to rubber-stamp AI output. You are being asked to use a tool that eliminates the most tedious part of the job - translating a clear specification into boilerplate code - so your time is spent on the parts that actually require your expertise: architecture, integration, debugging, and the judgment calls no specification can fully anticipate.
