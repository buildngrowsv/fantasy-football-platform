CLAUDE CODE PROMPT 1 — DATA LAYER: PROP INGESTION PIPELINE

This module handles all data collection from The Odds API. It runs as an AWS Lambda function triggered by EventBridge. It is the first step in the weekly pipeline and must be bulletproof — if this fails, nothing downstream runs.

# CLAUDE CODE PROMPT — PROP INGESTION PIPELINE
# Paste this entire block into Claude Code
# ============================================
 
Build a Python module called prop_ingestion.py that ingests NFL player
prop data from The Odds API and writes it to PostgreSQL.
 
ENVIRONMENT:
  Python 3.11
  AWS Lambda (512MB RAM, 15-minute timeout)
  Dependencies: requests, psycopg2-binary, boto3, python-dotenv
  Secrets: DB_URL and ODDS_API_KEY retrieved from AWS Secrets Manager at runtime
 
THE ODDS API ENDPOINTS TO CALL (in this order):
  1. GET /v4/sports/americanfootball_nfl/events
     Returns all games for the week with event IDs
 
  2. For each event: GET /v4/events/{eventId}/odds
     params: regions=us, markets=player_reception_yds,player_rush_yds,
             player_pass_yds,player_anytime_td,player_td_2+,player_td_3+,
             player_receptions,player_reception_longest,player_rush_attempts
     Returns: all books' lines for all player props in that game
 
  3. For alt lines: GET /v4/events/{eventId}/odds
     params: regions=us, markets=player_reception_yds_over_under (alternate)
     Fetch alt line rungs: 25,40,50,60,75,80,90,100,125,150,175,200 for yards
     Fetch alt reception rungs: 3,4,5,6,7,8,9,10
 
REQUIRED BEHAVIOR:
 
  1. DEDUPLICATION: Before inserting any prop, check if it already exists
     in raw_props for this player + game + prop_type + threshold + book_name.
     If it exists and the odds changed: UPDATE. If unchanged: skip.
     If new: INSERT.
 
  2. SUSPENSION DETECTION: At the start of each run, query raw_props for
     all props ingested in the previous run for this week's games.
     For any prop that was present in the previous run but is ABSENT from
     the current API response: set is_suspended = TRUE and pulled_at = NOW().
     This is a PRIORITY ALERT trigger. Write suspended prop IDs to a
     separate suspended_props_alert table for the notification dispatcher.
 
  3. OPENING LINE TRACKING: The first time a prop is seen for a game,
     record line_open and odds_open_over. Never overwrite these.
     Only current over_odds and under_odds update on subsequent pulls.
 
  4. STALE BOOK DETECTION: A book is stale if its line has not changed
     in 48+ hours while at least 2 other books have moved.
     Add a boolean column is_stale_book to raw_props updates.
     Stale books are excluded from consensus calculations downstream.
 
  5. RATE LIMITING: The Odds API has request limits.
     Add 0.5 second delay between event calls.
     On 429 response: wait 60 seconds, retry once. On second 429: log and skip.
 
  6. ERROR HANDLING:
     - Log all errors to a pipeline_errors table: (run_id, error_type,
       error_message, player_id if known, game_id if known, timestamp)
     - On DB connection failure: retry 3 times with exponential backoff
     - On API failure for a specific event: log and continue to next event
     - On complete API failure (5xx on /events call): abort run, send
       alert to SNS topic 'novapredict-pipeline-alerts'
 
  7. RUN METADATA: Write a pipeline_runs table entry for every execution:
     (run_id UUID, run_type, started_at, completed_at, props_inserted,
     props_updated, props_suspended, errors_count, status)
 
WRITE TESTS FOR:
  - Correct deduplication logic (update vs insert vs skip)
  - Suspension detection when a prop disappears
  - Opening line preservation (never overwritten)
  - Stale book flag logic
  - Rate limit retry behavior (mock the 429 response)
  - Error logging to pipeline_errors
  Use pytest. Mock all external calls (Odds API, AWS Secrets Manager, DB).

CLAUDE CODE PROMPT 2 — COMPUTATION: VIG REMOVAL ENGINE

The vig removal engine is the mathematical foundation of the entire platform. Every probability that flows into the CDF fitting, Monte Carlo, and projection system passes through this module. It must be exactly correct. Test coverage must be exhaustive.

# CLAUDE CODE PROMPT — VIG REMOVAL ENGINE
# ===========================================
 
Build a Python module called vig_removal.py with the following functions.
This module has no side effects — pure functions only. No database calls.
No external dependencies beyond standard math.
 
FUNCTION 1: american_to_raw_prob(odds: int) -> float
  Convert American odds to raw implied probability.
  Formula:
    If odds > 0:  P_raw = 100 / (odds + 100)
    If odds < 0:  P_raw = abs(odds) / (abs(odds) + 100)
    If odds == 0: raise ValueError('Odds cannot be zero')
  Return: float between 0.0 and 1.0
  Raise: ValueError if odds is 0 or if result is not in (0, 1)
 
FUNCTION 2: remove_vig_two_sided(
    over_odds: int, under_odds: int) -> tuple[float, float]
  Remove vig from a two-sided market (primary o/u lines).
  Formula:
    P_raw_over = american_to_raw_prob(over_odds)
    P_raw_under = american_to_raw_prob(under_odds)
    total = P_raw_over + P_raw_under
    P_novig_over = P_raw_over / total
    P_novig_under = P_raw_under / total
  Return: (P_novig_over, P_novig_under)
  REQUIRED: P_novig_over + P_novig_under must equal 1.0 (within 0.000001)
  REQUIRED: Both values must be in (0.0, 1.0) exclusive
  Raise: ValueError if either input odds is 0
 
FUNCTION 3: remove_vig_one_sided(
    odds: int,
    book_tier: str = 'PRIMARY'  # 'PRIMARY' | 'SECONDARY'
) -> float
  Remove vig from a one-sided market (alt lines, milestone props).
  Vig rates by tier:
    PRIMARY (DK, FD, BetMGM): vig_rate = 0.08
    SECONDARY (other books):  vig_rate = 0.10
  Formula:
    P_raw = american_to_raw_prob(odds)
    P_novig = P_raw / (1.0 + vig_rate)
  Return: float between 0.0 and 1.0
 
FUNCTION 4: compute_consensus_probability(
    probabilities: list[float],
    is_pinnacle: list[bool],
    is_stale: list[bool]
) -> tuple[float, int]
  Compute cross-book consensus no-vig probability.
  Rules:
    1. Exclude all books where is_stale[i] is True
    2. For Pinnacle (is_pinnacle[i] is True): include twice in the dataset
       (double-weight by duplication before computing median)
    3. Compute median of the remaining probabilities
  Return: (consensus_probability, active_book_count)
    active_book_count = count of non-stale books (Pinnacle counts as 1)
  REQUIRED: If active_book_count < 1: raise ValueError('No active books')
  REQUIRED: If active_book_count < 3: still compute but flag in return

# CLAUDE CODE PROMPT — MONTE CARLO PPR SIMULATOR
# ==================================================
 
Build a Python module called monte_carlo.py.
Dependencies: numpy, scipy. No database calls. Pure functions.
Performance requirement: 100,000 scenarios must complete in under 3 seconds
on a Lambda instance. Use vectorized numpy operations throughout.
DO NOT use Python loops for the simulation draws.
 
FUNCTION 1: sample_from_cdf(
    cdf_fn: callable,
    n_samples: int,
    stat_min: float = 0.0,
    stat_max: float = 400.0
) -> np.ndarray
  Draw n_samples from a distribution defined by a CDF function.
  Method: inverse CDF sampling (quantile function method)
    1. Draw n_samples uniform random numbers u ~ Uniform(0, 1)
    2. For each u, find x such that CDF_FN(x) = u
       (this is the inverse CDF / quantile function)
    3. Vectorize using scipy.optimize or a lookup table approach
  Return: np.ndarray of shape (n_samples,) containing stat values
  REQUIRED: All values must be >= stat_min and <= stat_max
  REQUIRED: No NaN values in output
 
FUNCTION 2: simulate_ppr(
    yards_cdf: callable,
    rec_cdf: callable,
    td_probs: dict,           # {'td_1plus': float, 'td_2plus': float, 'td_3plus': float}
    correlation_coeff: float, # 0.40 to 0.90, controls yards-receptions correlation
    n_scenarios: int = 100000,
    position: str = 'WR',     # 'WR' | 'RB' | 'TE' | 'QB'
    scoring: str = 'PPR'      # 'PPR' | 'HALF_PPR' | 'STANDARD'
) -> np.ndarray
  Run Monte Carlo simulation. Return array of n_scenarios PPR outcomes.
 
  STEP 1: Draw correlated yards and receptions samples
    Use Gaussian copula to enforce correlation_coeff between yards and rec:
    a. Generate correlated normal variables using numpy with Cholesky
       cov_matrix = [[1, corr], [corr, 1]]
       L = np.linalg.cholesky(cov_matrix)
       z = np.random.standard_normal((2, n_scenarios))
       correlated_z = L @ z
    b. Convert to uniform: u = scipy.stats.norm.cdf(correlated_z)
    c. Map through inverse CDFs:
       yards_samples = sample_from_cdf(yards_cdf, n_scenarios) using u[0]
       rec_samples = sample_from_cdf(rec_cdf, n_scenarios) using u[1]
 
  STEP 2: Draw TD samples
    Model TDs as discrete: 0, 1, 2, or 3
    P(0 TDs) = 1 - td_probs['td_1plus']
    P(1 TD)  = td_probs['td_1plus'] - td_probs['td_2plus']
    P(2 TDs) = td_probs['td_2plus'] - td_probs['td_3plus']
    P(3 TDs) = td_probs['td_3plus']
    Ensure all probabilities are non-negative (clamp if needed)
    Draw using np.random.choice with these probabilities
 
  STEP 3: Compute PPR for each scenario
    PPR scoring multipliers by position and format:
      rec_points: PPR=1.0, HALF_PPR=0.5, STANDARD=0.0
    ppr = (yards_samples / 10.0) + (rec_samples * rec_points) + (td_samples * 6.0)
    Ensure all values >= 0.0
 
  Return: np.ndarray of shape (n_scenarios,) containing PPR values
 
FUNCTION 3: extract_ppr_stats(ppr_array: np.ndarray) -> dict
  Extract all distribution statistics from Monte Carlo output.
  Return dict:
    floor_ppr:    np.percentile(ppr_array, 10)
    p25_ppr:      np.percentile(ppr_array, 25)
    median_ppr:   np.percentile(ppr_array, 50)
    p75_ppr:      np.percentile(ppr_array, 75)
    ceiling_ppr:  np.percentile(ppr_array, 90)
    boom_pct:     float(np.mean(ppr_array >= 20.0) * 100)
    bust_pct:     float(np.mean(ppr_array < 10.0) * 100)
    mean_ppr:     float(np.mean(ppr_array))
    std_ppr:      float(np.std(ppr_array))
 
FUNCTION 4: blend_distributions(
    market_stats: dict,      # output of extract_ppr_stats on market simulation
    internal_stats: dict,    # output of extract_ppr_stats on internal model
    market_confidence: float # 0.0 to 1.0
) -> dict
  Blend market and internal distributions percentile by percentile.
  Formula for each stat:
    blended_x = (market_x * market_confidence) + (internal_x * (1 - market_confidence))
  Apply to: floor_ppr, p25_ppr, median_ppr, p75_ppr, ceiling_ppr, boom_pct, bust_pct
  REQUIRED after blending:
    floor_ppr <= p25_ppr <= median_ppr <= p75_ppr <= ceiling_ppr
    If ordering is violated: sort and log a WARNING (do not raise)
    boom_pct and bust_pct must each be in [0.0, 100.0]
    boom_pct + bust_pct must be <= 100.0
  Return: dict with same keys as input stats dicts
 
WRITE TESTS FOR:
  simulate_ppr:
    Output shape is (n_scenarios,)
    All values >= 0.0
    No NaN values
    Mean is approximately median for symmetric distributions
    Performance: 100,000 scenarios in < 3 seconds
    Correlation: higher correlation_coeff produces higher yards-rec correlation
      in the output (measure with np.corrcoef)
 
  extract_ppr_stats:
    floor <= p25 <= median <= p75 <= ceiling
    boom_pct + bust_pct <= 100.0
    All values are Python floats (not numpy scalars)
 
  blend_distributions:
    market_confidence=1.0 -> output equals market_stats
    market_confidence=0.0 -> output equals internal_stats
    market_confidence=0.5 -> output is average of both

n8n workflows are built in the n8n UI, not in code. However, Claude Code can generate the JSON workflow definitions that you import directly into n8n. Each workflow below is specified as an engineering requirement — paste into Claude Code to generate the importable JSON.

# CLAUDE CODE PROMPT — N8N WORKFLOW DEFINITIONS
# =================================================
 
Generate n8n workflow JSON files for the NovaPredict orchestration layer.
Each workflow should be a complete, importable n8n workflow JSON.
Use n8n version 1.x node types.
 
WORKFLOW 1: weekly_pipeline.json
  Trigger: Schedule node - Tuesday 6:00 AM ET (cron: '0 6 * * 2')
  Steps:
    1. HTTP Request: POST to AWS Lambda 'weekly_data_pull' invoke URL
       Wait for completion (sync invoke). Timeout: 14 minutes.
    2. IF node: Check Lambda response statusCode == 200
       On success: proceed to step 3
       On failure: send Slack/email alert with error details, STOP
    3. HTTP Request: POST to AWS Lambda 'computation_engine' invoke URL
       Async invoke (don't wait). Fire and forget.
    4. Wait node: 10 minutes (allow computation to complete)
    5. HTTP Request: POST to AWS Lambda 'content_generation' invoke URL
       Sync invoke. Timeout: 14 minutes.
    6. IF node: Check content_generation response
       On success: send webhook to human_review_queue workflow
       On failure: send alert, log to Airtable error table
    7. Airtable node: Update pipeline_runs record with completion status
 
WORKFLOW 2: human_review_queue.json
  Trigger: Webhook (called by weekly_pipeline when content is ready)
  Steps:
    1. HTTP Request: GET list of draft profiles from PostgreSQL REST API
       Filter: status = 'DRAFT', created_at > NOW() - 2 hours
    2. Send Email via ConvertKit or SMTP:
       To: [editor email]
       Subject: 'NovaPredict: [N] player profiles ready for review'
       Body: links to each draft profile in the review interface
    3. Airtable node: Create review task record with draft count and deadline
 
WORKFLOW 3: sunday_inactives_sweep.json
  Trigger: Schedule - Sunday 7:30 AM ET (cron: '30 7 * * 0')
  Steps:
    1. HTTP Request: POST to Lambda 'inactives_sweep'
       Sync invoke. Timeout: 5 minutes.
    2. Code node (JavaScript): Parse Lambda response
       Extract: list of inactive player_ids
    3. IF node: inactive_count > 0
       On yes: proceed to step 4
       On no: log 'no inactives' to Airtable, STOP
    4. HTTP Request: POST to Lambda 'notification_dispatcher'
       Body: { type: 'INACTIVE_ALERT', player_ids: [list] }
    5. Airtable node: Log sweep results (count, player names, timestamp)
 
WORKFLOW 4: post_game_processing.json
  Trigger: Schedule - Monday 8:00 AM ET (cron: '0 8 * * 1')
  Steps:
    1. HTTP Request: POST to Lambda 'post_game_processing'
       Sync invoke. Timeout: 14 minutes.
    2. Code node: Parse response, extract miss classifications
    3. HTTP Request: POST to Lambda 'content_generation'
       Body: { type: 'ACCOUNTABILITY_REPORT', week: [current_week] }
    4. Airtable node: Update season accuracy metrics
    5. IF node: any VARIANCE_EVENT classifications present
       On yes: send alert email 'VARIANCE_EVENT requires human confirmation'
    6. Email node: Send 'Weekly processing complete' summary to admin
 
WORKFLOW 5: prop_suspension_alert.json
  Trigger: Webhook (called by prop_ingestion Lambda when suspension detected)
  Steps:
    1. Code node: Parse incoming webhook data
       Extract: player_id, player_name, prop_type, last_known_line
    2. HTTP Request: GET user_ids watching this player
       Query PostgreSQL: SELECT user_id FROM user_watchlists WHERE player_id = ?
    3. HTTP Request: POST to Lambda 'notification_dispatcher'
       Body: { type: 'PROP_SUSPENDED', player_id, user_ids: [list] }
    4. Airtable node: Log suspension event
 
For each workflow, generate:
  - Complete n8n JSON (importable via Settings > Import Workflow)
  - All nodes properly connected with correct node types
  - Error handling nodes on all HTTP Request nodes
  - Proper credential references (use placeholder names: 'PostgreSQL_NovaPredict',
    'AWS_Lambda_NovaPredict', 'Airtable_NovaPredict')

CLAUDE CODE PROMPT 8 — REACT NATIVE APP SHELL

# CLAUDE CODE PROMPT — REACT NATIVE APP SHELL
# ================================================
 
Build the React Native app shell for NovaPredict using Expo managed workflow.
This prompt covers: navigation structure, authentication, push notifications,
web view integration, and the five primary tab screens.
It does NOT cover individual feature screens (those are separate prompts).
 
SETUP:
  npx create-expo-app NovaPredict --template blank-typescript
  Dependencies to install:
    expo-router (file-based routing)
    expo-notifications
    expo-secure-store (for auth token storage)
    react-native-webview
    @react-navigation/bottom-tabs
    react-native-safe-area-context
    react-native-screens
 
APP ARCHITECTURE:
  Use expo-router with file-based routing.
  Root layout: app/_layout.tsx
  Auth check on startup: if no valid token, redirect to /auth/login
  Main app: app/(tabs)/_layout.tsx with 5 bottom tabs
 
AUTHENTICATION:
  - Login screen: email + password form
  - POST to https://api.novapredict.com/auth/login
  - Store JWT in expo-secure-store (key: 'nova_auth_token')
  - Token refresh: if API returns 401, attempt refresh then retry once
  - Logout: clear secure store, redirect to login
 
FIVE TAB SCREENS (bottom navigation):
 
  Tab 1: Home (icon: house)
    File: app/(tabs)/index.tsx
    Renders: personalized dashboard
    Data source: GET /api/home?userId={userId}
    Content: team watchlist alerts, current week top story,
             injury flags, challenge record widget, signal feed
    This is a NATIVE screen (not a web view)
 
  Tab 2: Players (icon: person.3)
    File: app/(tabs)/players.tsx
    Renders: WebView loading https://www.novapredict.com/players
    Pass auth token as cookie to the web view
    The website handles the full player database and search
    Individual player cards open as web views: /players/{playerId}
 
  Tab 3: Challenge (icon: trophy)
    File: app/(tabs)/challenge.tsx
    Renders: NATIVE screen
    Sub-screens (nested navigation):
      /challenge/pick-slate   - weekly pick interface
      /challenge/score-card   - Monday results
      /challenge/my-record    - season statistics
      /challenge/leaderboard  - public rankings (web view)
 
  Tab 4: Research (icon: magnifyingglass)
    File: app/(tabs)/research.tsx
    Renders: WebView loading https://www.novapredict.com/research
    Historical Intelligence Engine query bar lives on the website
    Waiver wire, trade analyzer are web view pages
 
  Tab 5: Reports (icon: doc.text)
    File: app/(tabs)/reports.tsx
    Renders: WebView loading https://www.novapredict.com/reports
    All content (accountability reports, player profiles, expert board) is web
 
PUSH NOTIFICATIONS:
  Register for push notifications on first launch (request permission)
  Store Expo push token in user profile via POST /api/users/push-token
  Handle incoming notifications:
    INACTIVE_ALERT -> navigate to /challenge/pick-slate with player highlighted
    PROP_SUSPENDED -> navigate to /players/{playerId}
    SCORE_CARD_READY -> navigate to /challenge/score-card
    EDGE_FINDER_BADGE -> navigate to /challenge/my-record
    CONTENT_READY -> navigate to /reports (weekly report)
    INJURY_UPDATE -> navigate to /players/{playerId}
    LINEUP_LOCK_REMINDER -> navigate to /challenge/pick-slate
 
WEB VIEW CONFIGURATION:
  All web views share these settings:
    javaScriptEnabled: true
    domStorageEnabled: true
    sharedCookiesEnabled: true
    thirdPartyCookiesEnabled: true
    Pass Authorization: Bearer {token} header to all novapredict.com requests
    onNavigationStateChange: intercept internal links to open natively vs externally
    Pull-to-refresh on all web views
 
REQUIRED: Build a useAuth hook that:
  - Reads token from secure store
  - Exposes: isAuthenticated, userId, subscriptionStatus, token
  - Handles token refresh automatically
  - Exposes logout function
 
REQUIRED: Build a useNotifications hook that:
  - Requests notification permission on mount
  - Registers push token with server
  - Adds notification received handler
  - Adds notification response handler (user tapped notification)
  - Handles navigation based on notification type
 
Write TypeScript throughout. Use proper types for all API responses.
The app should compile without TypeScript errors.
Write basic component tests for the auth hook and notification hook.

CLAUDE CODE PROMPT 9 — NEXT.JS WEBSITE

# CLAUDE CODE PROMPT — NEXT.JS WEBSITE
# =========================================
 
Build the Next.js 14 website for NovaPredict using the App Router.
This site serves both as the content destination (player profiles, reports)
and as the web view content for the React Native app.
 
SETUP:
  npx create-next-app@14 novapredict-web --typescript --tailwind --app
  Additional dependencies:
    @vercel/analytics
    next-auth (authentication shared with app)
    swr (data fetching with revalidation)
 
KEY PAGES AND THEIR SEO STRATEGY:
 
  / (homepage)
    Target keywords: fantasy football AI, fantasy football predictions
    Static generation. Revalidate every hour during season.
    Content: value proposition, accuracy numbers (fetched fresh), free CTA
 
  /accuracy (Expert Comparison Board)
    Target: fantasy football expert accuracy comparison
    PUBLIC — no auth required. This is the acquisition page.
    Static generation with 1-hour revalidation.
    Schema markup: FAQPage with common accuracy questions
 
  /players (Player Database)
    Target: [player name] fantasy football week [N]
    Auth required for full card. Free users see median only.
    /players/[playerId] - individual player card
    /players/[playerId]/week-[N] - weekly profile
    Dynamic routes with generateStaticParams for all current players
    ISR: revalidate = 3600 (hourly during season)
 
  /reports (Weekly Accountability Reports)
    Target: fantasy football weekly accuracy report
    /reports/week-[N]-[season] - individual week report
    Auth required for full report. Headline numbers public.
    Static generation. Revalidate Tuesday after report publishes.
 
  /research (Historical Intelligence Engine)
    Target: historical fantasy football stats analysis
    Auth required for query engine. 3 free queries per week for guests.
    Query results page: /research/results/[queryId]
    Query result pages are publicly accessible (good SEO — each result is a page)
 
SEO REQUIREMENTS:
  Every page must have:
    - Unique <title> tag with player name or report week
    - Meta description specific to page content
    - Open Graph tags for social sharing
    - Structured data (Schema.org) where applicable
    - Canonical URL
  Player profile pages: use Article schema
  Accuracy page: use FAQPage schema
  All pages: include breadcrumb schema
 
AUTHENTICATION:
  Use next-auth with JWT strategy
  Share the same JWT with the React Native app
  Provider: credentials (email/password)
  Middleware: protect /dashboard, /challenge, /research (beyond free limit)
  Free users: can access homepage, /accuracy, 3 /research queries/week,
    one /players/[playerId] profile per week (the featured player)
 
GOOGLE ANALYTICS + VERCEL ANALYTICS:
  Both installed. Every page view tracked.
  Track custom events:
    'paywall_hit' (when free user hits subscription gate)
    'query_submitted' (Historical Engine query)
    'profile_viewed' (player profile page load)
    'accuracy_board_viewed' (Expert Comparison Board)
 
PERFORMANCE REQUIREMENTS:
  Core Web Vitals targets:
    LCP < 2.5s
    CLS < 0.1
    FID < 100ms
  All player profile pages must be statically generated (not SSR)
  Images: use next/image with WebP format and lazy loading
 
Write TypeScript throughout. The site must build without errors.
Include basic Playwright tests for:
  - Homepage loads and shows accuracy numbers
  - /accuracy page loads without auth
  - /players page redirects to login if not authenticated
  - /research page shows query bar and free tier limit notice
